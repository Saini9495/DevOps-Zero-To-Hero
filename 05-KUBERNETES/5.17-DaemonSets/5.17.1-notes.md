# DaemonSets in Kubernetes

## What is a DaemonSet?

A DaemonSet ensures that a copy of a Pod runs on every node in the Kubernetes cluster.

Whenever a new node joins the cluster, the DaemonSet automatically creates a Pod on that node.

If a node is removed, the corresponding Pod is also removed.

---

## Why Do We Need DaemonSets?

Normally, Deployments schedule Pods on selected nodes.

However, some applications need to run on every node.

Examples:

- Log Collection Agents
- Monitoring Agents
- Security Agents
- Networking Components

DaemonSet solves this problem by ensuring one Pod per node.

---

## How DaemonSet Works?

Cluster

Node-1 → Fluentd Pod
Node-2 → Fluentd Pod
Node-3 → Fluentd Pod

When Node-4 joins:

Node-4 → Fluentd Pod (Automatically Created)

---

## Real Life Example

Imagine a company with multiple offices.

Every office requires:

- One Security Guard
- One CCTV System

Whenever a new office opens:

- Security Guard is assigned automatically.

DaemonSet works exactly the same way.

Node = Office

DaemonSet Pod = Security Guard

## DaemonSet YAML Structure

```yaml
apiVersion: apps/v1
kind: DaemonSet
metadata:
  name: fluentd
spec:
  selector:
    matchLabels:
      app: fluentd

  template:
    metadata:
      labels:
        app: fluentd

    spec:
      containers:
      - name: fluentd
        image: fluent/fluentd:v1.16
```

---

## How to Create DaemonSet?

```bash
kubectl apply -f daemonset.yml
```

---

## Verify DaemonSet

```bash
kubectl get daemonsets
```

Output:

NAME      DESIRED   CURRENT   READY
fluentd   3         3         3
```

---

## Check Pods

```bash
kubectl get pods -o wide
```

You will see one Pod running on each node.

---

## Describe DaemonSet

```bash
kubectl describe daemonset fluentd
```

---

## Delete DaemonSet

```bash
kubectl delete daemonset fluentd
```

---

## DaemonSet vs Deployment

| Feature | Deployment | DaemonSet |
|----------|-----------|------------|
| Pod Count | User Defined | One Per Node |
| Scaling | Manual/Auto | Automatic |
| Use Case | Applications | Node Agents |
| New Node Added | No New Pod Automatically | New Pod Created Automatically |

---

## Advantages

### Automatic Node Coverage

Runs on every node.

### Easy Management

Single configuration manages all nodes.

### Suitable for Monitoring

Perfect for node-level monitoring.

### Automatic Scheduling

New nodes automatically receive Pods.

---

## Limitations

### Resource Usage

Consumes resources on every node.

### Not Suitable for Applications

Use Deployments for web applications.

### Difficult Debugging

Issues may occur across multiple nodes.

---

## Interview Questions

### What is DaemonSet?

A DaemonSet ensures that a Pod runs on every node in a Kubernetes cluster.

---

### Difference Between Deployment and DaemonSet?

Deployment runs desired number of replicas.

DaemonSet runs one Pod on every node.

---

### Give Real World Use Cases of DaemonSet.

- Fluentd
- Filebeat
- Node Exporter
- Datadog Agent
- Calico

---

### What Happens When a New Node Joins?

DaemonSet automatically schedules a new Pod on that node.

---

### Can We Run More Than One Pod Per Node Using DaemonSet?

No.

By default, one Pod per node.
