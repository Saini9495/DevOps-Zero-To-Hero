# Affinity in Kubernetes

## What is Affinity?

Affinity is a scheduling feature in Kubernetes that controls on which node a Pod should be placed.

It provides more flexible scheduling compared to NodeSelector.

Using Affinity, we can define rules such as:

* Pod should run on a specific node.
* Pod should run with another Pod.
* Pod should avoid running with another Pod.
* Pod should run in a specific zone.

---

# Why Do We Need Affinity?

NodeSelector only supports exact label matching.

Example:

```yaml
nodeSelector:
  disktype: ssd
```

Problems:

* No advanced conditions
* No preference rules
* No Pod-to-Pod scheduling

Affinity solves these limitations.

---

# Types of Affinity

## 1. Node Affinity

Controls scheduling based on Node labels.

Example:

* Run only on SSD nodes
* Run only in Production nodes
* Run only in Zone-A

---

## 2. Pod Affinity

Controls scheduling based on other Pods.

Example:

Frontend Pod should run near Backend Pod.

Benefits:

* Lower latency
* Faster communication

---

## 3. Pod Anti-Affinity

Opposite of Pod Affinity.

Example:

Do not place two replicas of same application on same node.

Benefits:

* High Availability
* Fault Tolerance

---

# Affinity Architecture

```text
Node-1
 ├── Frontend Pod
 └── Backend Pod

Node-2
 └── Database Pod
```

Pod Affinity:

Frontend wants Backend nearby.

Scheduler places both on same node.

---

# Node Affinity

Node labels:

```bash
kubectl label node worker1 disktype=ssd
```

Node:

```yaml
labels:
  disktype: ssd
```

Pod:

```yaml
affinity:
  nodeAffinity:
```

Scheduler checks labels before scheduling Pod.

---

# Node Affinity Types

## Required During Scheduling

Mandatory Rule

If rule matches → Pod runs

If rule doesn't match → Pod remains Pending

```yaml
requiredDuringSchedulingIgnoredDuringExecution
```

---

## Preferred During Scheduling

Soft Rule

Scheduler tries to satisfy condition.

If condition not met, Pod still runs.

```yaml
preferredDuringSchedulingIgnoredDuringExecution
```

---

# Pod Affinity

Used when Pod wants to be scheduled near another Pod.

Example:

Frontend should run with Backend.

```text
Frontend Pod
      ↓
Backend Pod
```

Benefits:

* Faster communication
* Reduced network latency

---

# Pod Anti-Affinity

Used when Pods should not be scheduled together.

Example:

3 replicas of Nginx

Bad:

Node-1
├── nginx-1
├── nginx-2
└── nginx-3

If Node-1 fails, all Pods fail.

Good:

Node-1 → nginx-1

Node-2 → nginx-2

Node-3 → nginx-3

Achieved using Pod Anti-Affinity.

---

# Difference Between NodeSelector and Affinity

| Feature              | NodeSelector | Affinity |
| -------------------- | ------------ | -------- |
| Exact Match          | Yes          | Yes      |
| Expressions          | No           | Yes      |
| Preferred Rules      | No           | Yes      |
| Pod-Based Scheduling | No           | Yes      |
| Advanced Scheduling  | No           | Yes      |

Node Affinity:

Pod Affinity:

Frontend developer should sit near Backend developer.

Pod Anti-Affinity:

Two managers should not sit in same cabin.

# Production Use Cases

## Use Case 1

Database Pods only on SSD nodes.

```text
Database
     ↓
SSD Nodes
```

---

## Use Case 2

Frontend and Backend together.

```text
Frontend
     ↓
Backend
```

---

## Use Case 3

Multiple replicas separated.

```text
Replica-1 → Node-1
Replica-2 → Node-2
Replica-3 → Node-3
```

# Interview Questions

## What is Affinity in Kubernetes?

---

## What is Node Affinity?

Node Affinity schedules Pods based on Node labels.

---

## What is Pod Affinity?

Pod Affinity schedules Pods close to other Pods.

---

## What is Pod Anti-Affinity?

Pod Anti-Affinity prevents Pods from running together.

---

## Difference Between NodeSelector and Node Affinity?

NodeSelector supports only exact label matching.

Node Affinity supports advanced rules, expressions and preferences.

---

## What is RequiredDuringSchedulingIgnoredDuringExecution?

Mandatory scheduling rule.

Pod remains Pending if condition isn't satisfied.

---

## What is PreferredDuringSchedulingIgnoredDuringExecution?

Soft scheduling rule.

Pod can still run even if condition isn't satisfied.
