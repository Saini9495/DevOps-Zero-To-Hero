# Kubernetes Secrets

## What are Secrets?

Kubernetes Secrets are used to store sensitive information such as:

- Passwords
- API Keys
- Database Credentials
- SSH Keys
- Tokens

Instead of hardcoding confidential data inside Pod definitions, Kubernetes stores them securely in Secrets.

---

## Why Use Secrets?

Without Secrets:

```yaml
env:
  - name: DB_PASSWORD
    value: mypassword123
```

Problems:

- Visible to everyone
- Stored in GitHub
- Security Risk

With Secrets:

```yaml
env:
  - name: DB_PASSWORD
    valueFrom:
      secretKeyRef:
        name: mysql-secret
        key: password
```

---

## How Secrets Work

Developer
    ↓
Create Secret
    ↓
Kubernetes Secret Object
    ↓
Pod Accesses Secret
    ↓
Application Uses Credentials

---

## Secret Types

### 1. Opaque

Default Secret Type

Used for:

- Username
- Password
- API Keys

Example:

```yaml
type: Opaque
```

---

### 2. Docker Registry Secret

Used for private container registries.

Example:

```yaml
type: kubernetes.io/dockerconfigjson
```

---

### 3. TLS Secret

Used for HTTPS Certificates.

Example:

```yaml
type: kubernetes.io/tls
```

---

## Secret Data Encoding

Kubernetes stores Secret values in Base64 format.

Example:

Password:

```text
mypassword
```

Convert:

```bash
echo -n "mypassword" | base64
```

Output:

```text
bXlwYXNzd29yZA==
```

---

## Secret Manifest Structure

```yaml
apiVersion: v1
kind: Secret
metadata:
  name: mysql-secret
type: Opaque
data:
  username: cm9vdA==
  password: bXlwYXNzd29yZA==
```

---

## Using Secret as Environment Variable

```yaml
env:
- name: MYSQL_USER
  valueFrom:
    secretKeyRef:
      name: mysql-secret
      key: username
```

---

## Using Secret as Volume

```yaml
volumes:
- name: secret-volume
  secret:
    secretName: mysql-secret
```

Mounted inside container as files.

---

## Secret vs ConfigMap

| Feature | Secret | ConfigMap |
|----------|----------|----------|
| Sensitive Data | Yes | No |
| Password Storage | Yes | No |
| Base64 Encoding | Yes | No |
| Security Purpose | Yes | No |

---

## Real Life Example

Imagine a company office.

ConfigMap = Notice Board

Contains:

- Company policies
- Office timings

Secret = Locker

Contains:

- Cash
- Important documents
- Access cards

Everyone can read Notice Board.

Only authorized people can open Locker.

## Best Practices

### Don't Store Secrets in GitHub

Bad:

```yaml
password: mypassword123
```

Good:

```yaml
password: bXlwYXNzd29yZA==
```

---

### Enable Encryption at Rest

Protect Secret data inside etcd.

---

### Use RBAC

Restrict access to Secrets.

---

### Rotate Credentials

Change passwords regularly.

---

## Interview Questions

### What is a Kubernetes Secret?

A Kubernetes object used to store sensitive information securely.

---

### Difference Between Secret and ConfigMap?

Secret stores confidential information.

ConfigMap stores non-sensitive configuration.

---

### How are Secrets Stored?

Inside etcd in Base64 encoded format.

---

### Can Base64 be Decoded?

Yes.

Base64 is encoding, not encryption.

---

### How Can You Make Secrets More Secure?

- Encryption at Rest
- RBAC
- External Secret Manager
- Credential Rotation

---

## Summary

- Secrets store confidential data.
- Stored in Base64 format.
- Can be mounted as Environment Variables.
- Can be mounted as Volumes.
- More secure than hardcoding credentials.
