# Kubernetes Ingress

## What is Ingress?

Ingress is a Kubernetes resource that manages external access to services inside a cluster, typically HTTP and HTTPS traffic.

Instead of exposing every application using NodePort or LoadBalancer, Ingress provides a single entry point to route traffic to multiple services.

---

## Why Do We Need Ingress?

Without Ingress:

```text
User
  |
  |--> LoadBalancer --> App1
  |
  |--> LoadBalancer --> App2
  |
  |--> LoadBalancer --> App3
```

Problems:

* Multiple LoadBalancers required
* Increased cloud cost
* Difficult management
* Complex routing

With Ingress:

```text
User
  |
Ingress Controller
  |
  |--> App1 Service
  |
  |--> App2 Service
  |
  |--> App3 Service
```

Benefits:

* Single entry point
* Cost effective
* Host based routing
* Path based routing
* SSL/TLS support

---

## How Ingress Works

```text
Internet User
      |
      |
      ▼
Ingress Controller
      |
      |
 ┌────┴────┐
 │         │
 ▼         ▼

Service1  Service2
 │          │
 ▼          ▼

Pods      Pods
```

## Components of Ingress

### 1. Ingress Resource

Contains routing rules.

Example:

```yaml
apiVersion: networking.k8s.io/v1
kind: Ingress
```

---

### 2. Ingress Controller

Reads ingress rules and implements them.

Popular controllers:

* NGINX Ingress Controller
* Traefik
* HAProxy
* AWS ALB Controller

---

## Types of Routing

### Path Based Routing

```text
example.com/app1  --> app1-service
example.com/app2  --> app2-service
```

Example:

```yaml
paths:
- path: /app1
- path: /app2
```

---

### Host Based Routing

```text
app1.example.com --> app1-service

app2.example.com --> app2-service
```

Example:

```yaml
host: app1.example.com
host: app2.example.com
```

---

## Ingress vs Service

| Feature                | Service | Ingress |
| ---------------------- | ------- | ------- |
| Internal Communication | Yes     | No      |
| External Access        | Limited | Yes     |
| Path Routing           | No      | Yes     |
| Host Routing           | No      | Yes     |
| SSL Termination        | No      | Yes     |

## Ingress Controller Installation (Minikube)

Enable ingress:

```bash
minikube addons enable ingress
```

Verify:

```bash
kubectl get pods -n ingress-nginx
```

Expected Output:

```bash
ingress-nginx-controller
```

---

## Create Deployment

```bash
kubectl create deployment nginx --image=nginx
```

---

## Create Service

```bash
kubectl expose deployment nginx \
--port=80 \
--target-port=80
```

---

## Create Ingress

```yaml
apiVersion: networking.k8s.io/v1
kind: Ingress

metadata:
  name: nginx-ingress

spec:
  ingressClassName: nginx

  rules:
  - host: myapp.local

    http:
      paths:
      - path: /

        pathType: Prefix

        backend:
          service:
            name: nginx

            port:
              number: 80
```

## Real Life Example

Imagine a Shopping Mall.

Without Ingress:

Every shop has its own gate.

```text
Shop A --> Gate A
Shop B --> Gate B
Shop C --> Gate C
```

With Ingress:

```text
Main Gate
   |
   |
   ├── Shop A
   ├── Shop B
   └── Shop C
```

Ingress acts as the mall's main gate.

## Common Interview Questions

### What is Ingress?

Ingress is a Kubernetes API object that manages external HTTP and HTTPS access to services inside a cluster.

---

### Why use Ingress instead of LoadBalancer?

Ingress provides a single entry point and advanced routing features, reducing cloud costs.

---

### What is an Ingress Controller?

A component that watches ingress resources and implements routing rules.

---

### Difference Between Ingress and Service?

Service exposes applications internally while Ingress manages external traffic routing.

---

### What is Path Based Routing?

Routing traffic based on URL path.

Example:

```text
/app1 -> service1
/app2 -> service2
```

---

### What is Host Based Routing?

Routing traffic based on hostname.

Example:

```text
app1.example.com
app2.example.com
```
