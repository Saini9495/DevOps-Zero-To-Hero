# Taints and Tolerations

## What are Taints and Tolerations?

Taints and Tolerations are Kubernetes mechanisms used to control which Pods can be scheduled on specific Nodes.

By default, Kubernetes scheduler can place any Pod on any available Node.

Taints are applied to Nodes.

Tolerations are applied to Pods.

A Pod can only be scheduled on a tainted Node if it has a matching toleration.

---

## Why Do We Need Taints and Tolerations?

### Use Cases

* Reserve nodes for specific applications
* Dedicate nodes for databases
* Isolate production workloads
* Prevent resource-intensive applications from affecting other workloads
* Separate development and production environments

---

## Real Life Example

Imagine a hospital.

General Ward = Normal Node

ICU Ward = Tainted Node

Only patients with special permission can enter ICU.

Taint = ICU Entry Restriction

Toleration = ICU Permission Pass

Without the permission pass, patients cannot enter the ICU.

Similarly, Pods without tolerations cannot run on tainted nodes.

## Taint Syntax

kubectl taint nodes <node-name> key=value:effect

Example:

kubectl taint nodes worker-node env=production:NoSchedule

---

## Components of a Taint

env=production:NoSchedule

Key = env

Value = production

Effect = NoSchedule

---

## Types of Taint Effects

### 1. NoSchedule

New Pods without matching tolerations will NOT be scheduled.

Existing Pods continue running.

Example:

env=prod:NoSchedule

---

### 2. PreferNoSchedule

Scheduler tries to avoid scheduling Pods.

Not a strict rule.

If no better node is available, Pod may still be scheduled.

Example:

env=prod:PreferNoSchedule

---

### 3. NoExecute

New Pods are not scheduled.

Existing Pods are evicted immediately.

Example:

env=prod:NoExecute

---

## View Existing Taints

kubectl describe node <node-name>

Example:

kubectl describe node worker-node

Output:

Taints:
env=production:NoSchedule

---

## Add a Taint

kubectl taint nodes worker-node env=production:NoSchedule

---

## Remove a Taint

kubectl taint nodes worker-node env=production:NoSchedule-

Notice the minus (-) sign at the end.

---

## Tolerations

Tolerations allow Pods to run on tainted nodes.

Example:

tolerations:

* key: "env"
  operator: "Equal"
  value: "production"
  effect: "NoSchedule"

---

## How Scheduler Works

Step 1:

Scheduler checks available nodes.

Step 2:

Node contains taint.

env=production:NoSchedule

Step 3:

Scheduler checks Pod toleration.

Step 4:

If toleration matches:

Pod Scheduled

Else:

Pod Rejected

---

## Complete Example

Node Taint

kubectl taint nodes worker-node env=production:NoSchedule

Pod Toleration

tolerations:

* key: "env"
  operator: "Equal"
  value: "production"
  effect: "NoSchedule"

Result:

Pod successfully scheduled.

---

## Dedicated Database Node Example

Database Node:

kubectl taint nodes db-node database=mysql:NoSchedule

Only MySQL Pods should run there.

MySQL Pod:

tolerations:

* key: "database"
  operator: "Equal"
  value: "mysql"
  effect: "NoSchedule"

Result:

Only MySQL Pods can run on db-node.

---

## Master Node Taint

Control Plane nodes are automatically tainted.

Example:

node-role.kubernetes.io/control-plane:NoSchedule

This prevents user workloads from running on control plane nodes.

---

## Checking Taints in Cluster

kubectl get nodes

kubectl describe nodes

kubectl describe node worker-node | grep Taint

---

## Best Practices

1. Use dedicated nodes for databases.

2. Separate production workloads.

3. Avoid overusing taints.

4. Use labels and node selectors together.

5. Use NoExecute carefully because it can evict running Pods.

---

## Difference Between Node Selector and Taints

Node Selector

* Attraction Rule
* Pod chooses node

Taints

* Rejection Rule
* Node rejects pod

Node Selector = Invitation

Taint = Security Guard

---

## Difference Between Taints and Affinity

Node Affinity

* Preference based scheduling
* Flexible

Taints

* Restriction based scheduling
* Strict

---

## Interview Questions

### What are Taints and Tolerations?

Taints are applied to nodes to restrict scheduling.
Tolerations are applied to pods to allow scheduling on tainted nodes.

---

### What are the three taint effects?

1. NoSchedule
2. PreferNoSchedule
3. NoExecute

---

### Which taint effect removes existing Pods?

NoExecute

---

### What is the difference between NoSchedule and NoExecute?

NoSchedule prevents new Pods.

NoExecute prevents new Pods and removes existing Pods.

---

### Can a Pod run on a tainted node without a toleration?

No.

---

### Are Taints applied on Pods or Nodes?

Nodes.

---

### What is the purpose of Tolerations?

To allow Pods to run on tainted nodes.

---

### What is the command to remove a taint?

kubectl taint nodes <node-name> key=value:effect-

---

### What is the default taint on control plane nodes?

node-role.kubernetes.io/control-plane:NoSchedule
