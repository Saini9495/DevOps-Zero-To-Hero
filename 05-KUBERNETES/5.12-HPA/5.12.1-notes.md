# Horizontal Pod Autoscaler (HPA)

## What is HPA?

Horizontal Pod Autoscaler (HPA) is a Kubernetes resource that automatically increases or decreases the number of Pods in a Deployment, ReplicaSet, or StatefulSet based on resource utilization such as CPU or Memory.

Instead of manually scaling Pods, HPA continuously monitors metrics and adjusts replicas automatically.

---

## Why HPA is Needed?

Imagine an e-commerce website.

### Normal Traffic

100 Users
↓
2 Pods

### High Traffic

5000 Users
↓
10 Pods

### Low Traffic

50 Users
↓
2 Pods

Without HPA, scaling must be done manually.

With HPA, Kubernetes automatically scales Pods based on workload.

---

## How HPA Works?

1. Metrics Server collects CPU/Memory metrics.
2. HPA checks Pod utilization.
3. HPA compares actual utilization with target utilization.
4. If utilization is higher than target:

   * Increase Pods
5. If utilization is lower than target:

   * Decrease Pods

## Prerequisites

Before using HPA:

### 1. Metrics Server Must Be Installed

Check:

```bash
kubectl get pods -n kube-system
```

You should see:

```bash
metrics-server
```

---

### 2. Resource Limits Must Be Defined

Example:

```yaml
resources:
  requests:
    cpu: "100m"
  limits:
    cpu: "200m"
```

Without resource requests, HPA cannot calculate utilization.

---

## HPA Formula

Target CPU Utilization = 50%

Current CPU Utilization = 100%

Current Replicas = 2

Desired Replicas:

# Desired Replicas

Current Replicas × (Current Utilization ÷ Target Utilization)

Example:

2 × (100 ÷ 50)

= 4 Pods

Kubernetes automatically scales to 4 Pods.

## HPA vs VPA

| Feature      | HPA         | VPA                 |
| ------------ | ----------- | ------------------- |
| Scaling Type | Horizontal  | Vertical            |
| Action       | Add Pods    | Increase CPU/Memory |
| Downtime     | No          | Possible            |
| Usage        | Most Common | Less Common         |

---

## HPA vs Cluster Autoscaler

| Feature | HPA              | Cluster Autoscaler |
| ------- | ---------------- | ------------------ |
| Scales  | Pods             | Nodes              |
| Level   | Application      | Infrastructure     |
| Example | 2 Pods → 10 Pods | 2 Nodes → 5 Nodes  |

---

## Production Best Practices

* Always install Metrics Server
* Define resource requests and limits
* Set reasonable minReplicas
* Avoid aggressive scaling
* Monitor HPA events
* Use Prometheus + Grafana

## Interview Questions

### What is HPA?

Horizontal Pod Autoscaler automatically scales Pods based on resource utilization.

---

### Which metrics can HPA use?

* CPU
* Memory
* Custom Metrics
* External Metrics

---

### Does HPA scale Nodes?

No.

HPA scales Pods only.

Cluster Autoscaler scales Nodes.

---

### What is required before using HPA?

Metrics Server and resource requests.

---

### Difference between HPA and VPA?

HPA adds Pods.

VPA increases CPU/Memory of existing Pods.

---

### Can HPA work without Metrics Server?

No.

Metrics Server is mandatory.

---

## Summary

HPA is one of the most important Kubernetes resources used in production environments.

It continuously monitors CPU/Memory utilization and automatically scales application Pods up or down to maintain performance and optimize resource usage.
