# RBAC (Role Based Access Control)

## What is RBAC?

RBAC (Role Based Access Control) is a Kubernetes security mechanism used to control who can access Kubernetes resources and what actions they can perform.

RBAC allows administrators to define permissions for users, groups, or service accounts.

---

## Why RBAC?

Without RBAC every authenticated user may have excessive permissions which can create security risks.

RBAC helps in:

* Access Control
* Security
* Least Privilege Principle
* Multi-Team Environment
* Compliance Requirements

---

## RBAC Components

### 1. Role

A Role defines permissions within a specific namespace.

Example:

* Read Pods
* Create Deployments
* Delete Services

Role is namespace scoped.

---

### 2. ClusterRole

A ClusterRole defines permissions across the entire cluster.

Example:

* Access Nodes
* Access Persistent Volumes
* Manage Namespaces

ClusterRole is cluster scoped.

---

### 3. RoleBinding

A RoleBinding connects:

User / Group / ServiceAccount → Role

Namespace specific.

---

### 4. ClusterRoleBinding

A ClusterRoleBinding connects:

User / Group / ServiceAccount → ClusterRole

Cluster wide.

---

## RBAC Architecture

User
↓
RoleBinding
↓
Role
↓
Permissions

OR

User
↓
ClusterRoleBinding
↓
ClusterRole
↓
Permissions

# Role vs ClusterRole

| Feature           | Role | ClusterRole |
| ----------------- | ---- | ----------- |
| Namespace Scoped  | Yes  | No          |
| Cluster Wide      | No   | Yes         |
| Access Pods       | Yes  | Yes         |
| Access Nodes      | No   | Yes         |
| Access Namespaces | No   | Yes         |

# Role Example

This role allows reading pods.

```yaml
apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata:
  namespace: dev
  name: pod-reader

rules:
- apiGroups: [""]
  resources: ["pods"]
  verbs: ["get", "watch", "list"]
```

---

# RoleBinding Example

Bind the above role to a user.

```yaml
apiVersion: rbac.authorization.k8s.io/v1
kind: RoleBinding
metadata:
  name: read-pods
  namespace: dev

subjects:
- kind: User
  name: john
  apiGroup: rbac.authorization.k8s.io

roleRef:
  kind: Role
  name: pod-reader
  apiGroup: rbac.authorization.k8s.io
```

---

# ClusterRole Example

```yaml
apiVersion: rbac.authorization.k8s.io/v1
kind: ClusterRole
metadata:
  name: pod-reader-cluster

rules:
- apiGroups: [""]
  resources: ["pods"]
  verbs: ["get", "list", "watch"]
```

---

# ClusterRoleBinding Example

```yaml
apiVersion: rbac.authorization.k8s.io/v1
kind: ClusterRoleBinding
metadata:
  name: pod-reader-binding

subjects:
- kind: User
  name: john
  apiGroup: rbac.authorization.k8s.io

roleRef:
  kind: ClusterRole
  name: pod-reader-cluster
  apiGroup: rbac.authorization.k8s.io
```

# Service Account Example

Applications running inside pods usually use Service Accounts.

```bash
kubectl create serviceaccount app-user
```

Check:

```bash
kubectl get serviceaccounts
```

---

# Verify Permissions

Check what a user can do:

```bash
kubectl auth can-i get pods
```

Check for another user:

```bash
kubectl auth can-i delete pods --as=john
```

Example Output:

```bash
yes
```

or

```bash
no
```

---

# Best Practices

1. Follow Least Privilege Principle.
2. Never grant cluster-admin unnecessarily.
3. Use Roles whenever possible.
4. Use ClusterRoles only when required.
5. Regularly audit permissions.
6. Use Service Accounts for applications.

# Common Interview Questions

## What is RBAC?

RBAC is a Kubernetes authorization mechanism used to control access to cluster resources.

---

## Difference Between Role and ClusterRole?

Role works within a namespace.

ClusterRole works across the entire cluster.

---

## Difference Between RoleBinding and ClusterRoleBinding?

RoleBinding grants namespace level access.

ClusterRoleBinding grants cluster wide access.

---

## What is a Service Account?

A Service Account is an identity used by applications running inside Kubernetes Pods.

---

## How Do You Check User Permissions?

```bash
kubectl auth can-i
```

Example:

```bash
kubectl auth can-i create deployments
```

---

## What is Cluster Admin?

Cluster Admin is the highest privilege role in Kubernetes that provides complete control over the cluster.

---

## What Principle Should Be Followed While Creating Roles?

Least Privilege Principle.
Only provide permissions that are actually required.
