# Node Selector

## What is Node Selector?

Node Selector is the simplest way to control where a Pod gets scheduled in a Kubernetes cluster.

It allows Pods to run only on nodes that have specific labels.

Kubernetes Scheduler checks the node labels and schedules the Pod on the matching node.

---

## Why Do We Need Node Selector?

By default Kubernetes can schedule Pods on any available node.

Sometimes we want:

* Database Pods on High Memory Nodes
* Monitoring Pods on Dedicated Nodes
* GPU Workloads on GPU Nodes
* Production Workloads on Production Nodes

Node Selector helps achieve this.

---

## How Node Selector Works?

### Step 1: Add Label to Node

```bash
kubectl label nodes worker-node-1 disktype=ssd
```

Verify:

```bash
kubectl get nodes --show-labels
```

Output:

```text
worker-node-1   disktype=ssd
worker-node-2   disktype=hdd
```

---

### Step 2: Add Node Selector in Pod

```yaml
nodeSelector:
  disktype: ssd
```

Scheduler searches for nodes having:

```text
disktype=ssd
```

and deploys the Pod there.

## Real Life Example

Imagine a delivery company.

You have three warehouses:

```text
Warehouse-1 -> Cold Storage
Warehouse-2 -> Electronics
Warehouse-3 -> Furniture
```

If an order requires cold storage, it should only go to Warehouse-1.

Node Labels = Warehouse Type

Node Selector = Delivery Rule

Pod = Package

---

## Advantages

* Easy to configure
* Lightweight
* Useful for small clusters
* Better workload isolation

---

## Limitations

* Supports only exact matching
* No complex conditions
* Cannot use OR conditions
* Less flexible than Node Affinity

---

## Node Selector vs Node Affinity

| Feature                    | Node Selector  | Node Affinity  |
| -------------------------- | -------------- | -------------- |
| Easy to Use                | Yes            | Medium         |
| Exact Match                | Yes            | Yes            |
| Expressions                | No             | Yes            |
| Advanced Scheduling        | No             | Yes            |
| Recommended for Production | Small Clusters | Large Clusters |

## Common Use Cases

### SSD Nodes

```yaml
nodeSelector:
  disktype: ssd
```

### GPU Nodes

```yaml
nodeSelector:
  gpu: "true"
```

### Production Nodes

```yaml
nodeSelector:
  environment: production
```

### Monitoring Nodes

```yaml
nodeSelector:
  monitoring: "true"
```

## Troubleshooting

### Pod Stuck in Pending State

Check:

```bash
kubectl describe pod nginx
```

Possible Output:

```text
0/2 nodes are available:
node(s) didn't match Pod's node selector.
```

Reason:

Node label does not match.

---

### Verify Labels

```bash
kubectl get nodes --show-labels
```

---

### Check Specific Node Labels

```bash
kubectl describe node worker-node-1
```

## Interview Questions

### What is Node Selector?

Node Selector is a Kubernetes scheduling mechanism that allows Pods to run only on nodes with specific labels.

---

### Why is Node Selector used?

It helps place workloads on specific nodes based on hardware, environment, or business requirements.

---

### What is the difference between Node Selector and Node Affinity?

Node Selector supports only exact matching, while Node Affinity supports advanced scheduling rules and expressions.

---

### What happens if no node matches the selector?

The Pod remains in Pending state until a matching node becomes available.

---

### How do you check node labels?

```bash
kubectl get nodes --show-labels
```

---

## Key Takeaways

* Node Selector is the simplest scheduling mechanism.
* Works using node labels.
* Supports exact matching only.
* Pod stays Pending if no matching node exists.
* Node Affinity is the advanced version of Node Selector.
